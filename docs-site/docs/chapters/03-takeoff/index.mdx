> *"Takeoff is procedural. Follow the checklist. This is where AI earns its keep."*

**Estimated time:** ~30 min | **AI skill focus:** Boilerplate generation | **Branch:** `chapter-3-takeoff`

**Anti-pattern:** Backseat Driver — over-delegating trivial code, using AI for things faster to type yourself.

---

## Section 1: Boilerplate is AI's Sweet Spot

Not all code is created equal. Some code requires deep thought — business logic, state management, algorithm design. Other code is structural — routing definitions, navigation shells, placeholder screens, wiring files together.

The structural code is boilerplate. It follows well-known patterns, has little room for creativity, and takes time without requiring much thinking. This is where AI assistance provides the best return on investment.

**Why boilerplate is low risk:**

Routing tables have a known structure. Navigation shells follow documented patterns. Placeholder screens are identical except for the title text. Theme wiring is mechanical. If the AI makes a mistake in boilerplate, you catch it immediately because the app does not compile or the navigation does not work. Compare this with a subtle state management bug that only appears when you log out and back in.

**The strategy for this chapter:** Let AI generate the structure, then manually adjust the details. We call this the "Scaffold and Refine" pattern.

---

## Section 2: The "Scaffold and Refine" Pattern

Instead of writing a perfect prompt that generates perfect code, use a two-step approach:

**Step 1 — Scaffold.** Give the AI a prompt that describes the overall structure. Accept the output, knowing it will be roughly 80% correct.

**Step 2 — Refine.** Read through the generated code and fix the details yourself. Adjust route paths, fix import paths, change icon choices, tweak naming conventions to match your project.

This pattern works because:

You do not waste time crafting the perfect prompt. The 80% the AI gets right would have taken you 15 minutes to type. The 20% you fix yourself takes 2 minutes and ensures the code matches your exact requirements. You understand every line because you reviewed it all.

The Scaffold and Refine pattern is the opposite of the Fly-by-Night anti-pattern from Chapter 1. You are not accepting blindly — you are using AI for the tedious parts and applying your judgement for the details.

## Section 3: Build the Router

PilotBank needs a `GoRouter` configuration with authentication-aware redirects. This is a textbook AI task — the structure is well-defined, the patterns are documented, and the details are specific to your app.

Create `lib/routing/router.dart` with:

1. A `routerProvider` using Riverpod `Provider` that watches `authProvider`
2. A `GoRouter` with `initialLocation: '/login'`
3. A redirect function that sends unauthenticated users to `/login` and redirects authenticated users away from `/login` to `/dashboard`
4. Routes: `/login` (standalone), then a `ShellRoute` wrapping `/dashboard`, `/transactions/:accountId`, `/transfer`, `/notifications`, `/settings`
5. Each route should use its corresponding screen widget

You need to import all screen files, the auth provider, and the GoRouter and Riverpod packages. Getting the redirect logic right is the tricky part — you need to handle both directions (not logged in trying to access a protected route, and logged in trying to access the login page).

  </Manual>

**With your copilot (~5 min):**

Review your `router.dart`:

Does the `routerProvider` use `ref.watch(authProvider)` to react to auth changes? Does the redirect handle **both** directions — unauthenticated users are sent to login, and authenticated users are redirected away from login?

Does the redirect return `null` when no redirect is needed? (Returning `null` means "stay on the current route.") Is `/login` outside the `ShellRoute`? The login screen should not have the bottom navigation bar.

Is the `ShellRoute` builder correct? It should be `(context, state, child) => ShellScreen(child: child)`. Does the transactions route correctly extract `accountId` from `state.pathParameters`?

## Section 4: When NOT to Use AI

Now that you have seen AI handle boilerplate efficiently, let's talk about when it is faster to type code yourself. This is the **Backseat Driver** anti-pattern — over-delegating trivial code to the AI.

### Things faster to type yourself

**One-line widgets.** Typing `const SizedBox(height: 16)` takes two seconds. Waiting for an AI suggestion, reading it, and accepting it takes longer. Your fingers are faster than the feedback loop for trivial code.

**Simple conditionals.** A basic `if/else` is faster to type than to prompt for. The AI adds no value on code you can write without thinking.

**Import statements.** Your IDE's auto-import is faster and more accurate than asking the AI. Press the quick-fix shortcut and let the IDE handle it.

**Variable declarations.** `final name = 'Alex';` does not need AI assistance.

### The decision rule

**AI saves time on 20+ line structures.** For 1-3 lines, your fingers are faster than the feedback loop. For 4-20 lines, it depends on complexity. For 20+ lines of structured, repetitive code, AI is almost always faster.

The Backseat Driver wastes time by delegating everything — including the trivial parts — to the AI. They wait for suggestions on `const SizedBox(height: 16)` instead of just typing it. They prompt for import statements instead of using auto-import. They ask the AI to write `return null;` instead of typing five characters.

## Section 5: Wiring It Together

The last step is connecting the router to the app. This requires updating two existing files.

### Update main.dart

Open `lib/main.dart`. You need to wrap the app in a `ProviderScope` so that Riverpod providers are available throughout the widget tree.

The change is small: import `flutter_riverpod`, and wrap `PilotBankApp()` with `ProviderScope`. This is a one-line structural change — a good example of "Scaffold and Refine" where the scaffold is tiny.

```dart
import 'package:flutter/material.dart';
import 'package:flutter_riverpod/flutter_riverpod.dart';
import 'app.dart';

void main() {
  WidgetsFlutterBinding.ensureInitialized();
  runApp(
    const ProviderScope(
      child: PilotBankApp(),
    ),
  );
}
```

### Update app.dart

Open `lib/app.dart`. The app currently uses `MaterialApp` with a `home` parameter. You need to change it to `MaterialApp.router` with a `routerConfig` that reads the GoRouter from the provider.

This means `PilotBankApp` needs to become a `ConsumerWidget` so it can access `ref.watch(routerProvider)`. You also need to remove the `_WelcomeScreen` class since the router now controls which screen appears.

```dart
import 'package:flutter/material.dart';
import 'package:flutter_riverpod/flutter_riverpod.dart';
import 'routing/router.dart';
import 'theme/app_theme.dart';

class PilotBankApp extends ConsumerWidget {
  const PilotBankApp({super.key});

  @override
  Widget build(BuildContext context, WidgetRef ref) {
    final router = ref.watch(routerProvider);

    return MaterialApp.router(
      title: 'PilotBank',
      debugShowCheckedModeBanner: false,
      theme: AppTheme.light(),
      darkTheme: AppTheme.dark(),
      themeMode: ThemeMode.system,
      routerConfig: router,
    );
  }
}
```

## Section 6: The Anti-Pattern — Backseat Driver

Chapter 3's anti-pattern is the **Backseat Driver**.

A Backseat Driver delegates everything to the AI, including tasks that are faster to do by hand. They prompt for one-line widgets, ask the AI to write import statements, and wait for suggestions on code they could type in seconds.

The result is slower development, not faster. The overhead of the AI feedback loop — writing a prompt, waiting for a response, reading the suggestion, deciding whether to accept — only pays off when the generated code saves more time than the loop costs.

**Signs you are a Backseat Driver:**

You ask the AI to generate `const SizedBox(height: 16)`. You wait for an inline suggestion instead of typing a three-line widget. You prompt for every import statement instead of using your IDE's auto-import. You use the chat panel to ask "how do I return null from a function."

**The fix:** Develop a sense for the complexity threshold. Below 5 lines of straightforward code, type it. Above 20 lines of structural code, prompt for it. In between, use your judgement. The goal is speed, not purity — use whichever method is faster for each piece of code.
