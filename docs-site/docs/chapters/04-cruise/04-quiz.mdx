<QuizQuestion
  question="Why does the LoginScreen use ConsumerStatefulWidget instead of ConsumerWidget?"
  options={[
    "Because ConsumerWidget does not support Riverpod",
    "Because it needs both Riverpod access (ref) and mutable state for TextEditingControllers that must be disposed",
    "Because ConsumerStatefulWidget is always preferred over ConsumerWidget",
    "Because the login screen has animations that require a State object"
  ]}
  correctIndex={1}
  explanation="The login screen needs ref.watch() and ref.read() for Riverpod (which ConsumerWidget provides), but it also needs TextEditingControllers that must be created in initState and disposed in dispose(). />

<QuizQuestion
  question="In the _submit() method, why does the code use ref.read(authProvider.notifier) instead of ref.watch(authProvider.notifier)?"
  options={[
    "read is faster than watch for provider access",
    "watch cannot access the notifier, only the state",
    "read is for one-time access in event handlers; watch is for reactive rebuilds in the build method",
    "There is no difference — both work the same way in this context"
  ]}
  correctIndex={2}
  explanation="ref.read() accesses a provider's value once without subscribing to changes. It is correct for event handlers like button presses. ref.watch() subscribes to the provider and triggers a rebuild when the value changes — it should only be used inside the build() method. Using watch in a callback is a Riverpod anti-pattern that can cause unexpected rebuilds."
/>

<QuizQuestion
