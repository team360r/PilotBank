---

## Section 1: Describing UI to AI

Building a screen with AI is not the same as describing it to a designer. Designers interpret intent. AI implements instructions. The distinction matters because vague descriptions produce vague results.

When you describe a screen to an AI coding assistant, you need to be explicit about four things:

**Layout structure.** Tell the AI which layout widgets to use — `Column`, `Row`, `ListView`, `Stack`. Do not say "put these things on the screen" and expect the AI to choose the right layout. Specify the hierarchy. A login screen is a `Column` inside a `SingleChildScrollView` with horizontal padding. A dashboard is a `ListView` with a header and repeating cards.

**Widget types.** Name the exact Material widgets. Say `TextField` not "input field." Say `ElevatedButton` not "button." Say `Card` with `InkWell` not "clickable card." The more precise your widget vocabulary, the more accurate the output.

**Spacing and padding.** Numbers matter. "Add some padding" produces inconsistent results. "Add 24px horizontal padding and 16px between fields" produces consistent results. If you care about the layout, specify the values.

**Component names.** Reference Material Design components by their Flutter class names. `AppBar`, `BottomNavigationBar`, `Scaffold`, `ListTile`. These are well-represented in training data, so the AI understands them precisely.

AI is not a design tool — it builds what you describe. If your description is vague, the UI will be vague.

## Section 2: Iterative Refinement

No prompt produces a perfect screen on the first try. Expect to iterate. This is not a failure — it is the process.

The generate-preview-reprompt loop works like this:

**Step 1 — Generate.** Write a prompt describing the screen structure. Accept the output.

**Step 2 — Hot reload.** Save the file and look at the result on your device or emulator. What is wrong? What is close but not right?

**Step 3 — Re-prompt with corrections.** Be specific. Not "make it look better" but "move the button below the text fields" or "add 16px padding around the card content" or "change the icon from star to chevron_right."

**Step 4 — Repeat.** Each cycle gets the screen closer. Two to three iterations is typical. More than five iterations means your initial prompt was too vague — start over with a more specific description.

The key insight is that corrections are cheap. A follow-up prompt like "add a SizedBox(height: 16) between the two TextFields" takes seconds to write and the AI applies it immediately. You do not need to get everything right in one prompt.

## Section 3: Build the Login Screen

The login screen is the first real screen in PilotBank. It needs to interact with the auth provider, manage its own text controller state, and display error messages.

Create a `ConsumerStatefulWidget` in `lib/screens/login_screen.dart` with:

8. An `ElevatedButton` with text "Sign in" styled with `AppColors.primary` background and white foreground
9. A `TextButton` saying "Forgot password?" below the sign-in button
10. An `onChanged` callback on both fields that calls `clearError()` when the user starts typing after an error

**With your copilot (~5 min):**

Import flutter_riverpod, auth_provider from ../providers/auth_provider.dart, and app_colors from ../theme/app_colors.dart. Create two TextEditingControllers for email and password, dispose them properly. The body should be a SafeArea wrapping a SingleChildScrollView with 24px horizontal padding. Add a Column with: a flight icon (size 64, AppColors.primary), 'PilotBank' heading text, 'Sign in to your account' subtitle, then a TextField with labelText 'Email address' and email keyboard type, a TextField with labelText 'Password' and obscureText true. Both text fields should call clearError() on change when there is an error showing."

  </AIAssisted>
  <CrossCheck>

Review your `login_screen.dart`:

Does the class extend `ConsumerStatefulWidget` (not `StatelessWidget` or `StatefulWidget`)? It needs both `ref` for Riverpod and `State` for the text controllers.

Does the `dispose()` method call `.dispose()` on **both** controllers before calling `super.dispose()`? Memory leaks from undisposed controllers are a common AI omission.

Does `_submit()` use `ref.read(authProvider.notifier).login(...)` with **read**, not watch? Using `watch` inside a callback would be incorrect — `read` is for one-time access in event handlers.

Is the error message conditionally shown only when `authState.error != null`? Does it use `AppColors.error` for the text color?

Does the `ElevatedButton` call `_submit` (not `_submit()`)? Passing the method reference is correct. Calling it with parentheses would invoke it during build.

Do the `onChanged` callbacks clear the error? This is a usability detail — users should not see a stale error message while correcting their input.

  </CrossCheck>
</DualPath>

## Section 4: Build the Dashboard and Account Card

The dashboard is the main screen after login. It shows a greeting and a list of account cards. Each card displays the account name, balance, and an icon for the account type. Tapping a card navigates to the transactions screen for that account.

This exercise requires two files: a reusable `AccountCard` widget and the `DashboardScreen` that uses it.

## Section 5: The Anti-Pattern — Armchair Architect

Chapter 4's anti-pattern is the **Armchair Architect**.

The Armchair Architect expects AI to make design decisions. They write prompts like "make a beautiful login page" or "create a modern-looking dashboard" and expect the AI to produce something visually polished. Then they are disappointed when the result looks generic or mismatched.

Here is the difference between a bad prompt and a good prompt:

**Bad prompt:**

> "Create a beautiful, modern login page for a banking app. Make it look professional and clean."

This prompt tells the AI nothing useful. "Beautiful" is subjective. "Modern" is vague. "Professional and clean" could describe a thousand different designs. The AI will produce something generic — probably a centred column with default Material styling — because it has no specific instructions to follow.

**Good prompt:**

> "Create a login screen with a SafeArea wrapping a SingleChildScrollView. 24px horizontal padding. Column with: a flight icon (size 64, primary colour), 'PilotBank' as headlineMedium bold, 'Sign in to your account' as bodyLarge in textSecondary. Two TextFields with outlined borders — email with labelText 'Email address' and password with labelText 'Password' and obscureText. An ElevatedButton in primary colour with white text saying 'Sign in'. A TextButton saying 'Forgot password?' below."

This prompt produces exactly what you described because it IS exactly what you described. There is no ambiguity for the AI to resolve. Every widget, every spacing value, every text string is specified.

**The underlying principle:** AI implements. You design. The design decisions — what goes where, what colour, what size, what text — are yours. The AI's job is to turn those decisions into code. If you delegate the design decisions to the AI, you get the AI's taste, which is "whatever was most common in the training data."
