<QuizQuestion
  question="A StateNotifier updates state with 'state.isLoggedIn = true'. What is wrong?"
  options={[
    "Nothing — this is the correct way to update state in Riverpod",
    "The property name should be camelCase",
    "Direct mutation does not trigger Riverpod's rebuild mechanism — the UI will not update",
    "You should use setState() instead of direct assignment"
  ]}
  correctIndex={2}
  explanation="Riverpod's StateNotifier detects changes by comparing old and new state references. Mutating a property on the existing state object does not create a new reference, so Riverpod does not know the state changed. The correct approach is state = AuthState(isLoggedIn: true) — creating a new object."
/>

<QuizQuestion
  question="When should you use a simple Provider instead of a StateNotifierProvider?"
  options={[
    "When the data needs to be updated by user actions",
    "When the data is read-only and never changes after initialisation",
    "When working with lists instead of single objects",
    "Always — Provider is the preferred type for all cases"
  ]}
  correctIndex={1}
  explanation="A simple Provider is the right choice for data that is read-only and never changes. If the data needs to change in response to user actions, you need a StateNotifierProvider with explicit update methods. Using the wrong type leads to either unnecessary complexity or missing functionality."
/>

<QuizQuestion
  question="Why is it important to check that a list-based StateNotifier creates new list objects instead of mutating in place?"
  options={[
    "Creating new lists uses less memory than modifying existing ones",
    "Dart does not support in-place list modification",
    "Riverpod compares by reference — if the list reference does not change, the UI does not rebuild",
    "The Dart analyzer will show a warning for in-place mutation"
  ]}
  correctIndex={2}
  explanation="Riverpod's StateNotifier compares old and new state by reference. If you mutate a list in place (e.g., state[0].read = true), the list reference stays the same, so Riverpod thinks nothing changed and does not rebuild the UI. You must create a new list with new objects."
