## Section 1: AI for Dartdoc Comments

Dartdoc comments are the documentation strings that appear above classes, methods, and properties in Dart code. They use the `///` syntax and are displayed in IDEs, documentation tools, and API references.

AI is good at generating dartdoc comments because:
- **It can infer purpose from code.** A method called `markAllRead` that iterates a list and sets `read: true` is clearly documented as "Marks all notifications as read."
- **It follows the dartdoc format.** Triple-slash comments, parameter descriptions with `[paramName]`, return value descriptions — these follow a well-documented pattern.
- **It fills in what you skip.** Most developers skip documentation for "obvious" methods. AI generates it consistently, even for simple getters.

AI is bad at dartdoc comments because:
- **It describes what it THINKS the code does, not what it actually does.** If a method has a bug, the AI documents the intended behaviour, not the actual behaviour.
- **It invents context.** A method called `login` might be documented as "Authenticates the user against the remote server" when it actually checks hardcoded credentials in mock data.
- **It copies patterns from training data.** Generic descriptions like "Creates an instance of [ClassName]" add no value — the reader already knows that from the constructor syntax.

The rule: AI generates the first draft. You verify every comment against the actual code. This takes less time than writing from scratch, but more time than copy-pasting without reading.

## Section 3: AI for Commit Messages and PR Descriptions

Beyond code documentation, AI is useful for another kind of writing: commit messages and pull request descriptions. These are communication artefacts — they tell other developers what changed and why.

**Commit messages:**

A conventional commit message has a type, optional scope, and description:

```
feat(transactions): add filter chips and date-grouped transaction list
```

You can prompt AI to generate these:

> "Summarise the following diff as a conventional commit message. Use the format: type(scope): description. Types: feat, fix, refactor, test, docs, chore. Keep the description under 72 characters."

AI handles this well because commit message conventions are well-documented and the diff provides all the context. The cross-check is simple: does the message accurately describe what changed?

**PR descriptions:**

Pull request descriptions need more structure. A useful template:

```markdown
## What changed
- Added transactions screen with filter chips
- Extracted TransactionTile into reusable widget

## Why
- Users need to view and filter transaction history per account

## How to test
1. Navigate to Dashboard
2. Tap any account card
3. Verify transactions appear grouped by date
4. Test each filter chip
```

> "Generate a PR description for this branch. Use sections: What changed (bullet points), Why (one sentence), How to test (numbered steps). Base it on the commit history."

The cross-check: read the "How to test" section and actually follow the steps. If a step does not work, the AI described what it thought the code does, not what it actually does.

## Section 4: The Anti-Pattern — Radio Silence

Chapter 8's anti-pattern is **Radio Silence**.

In aviation, radio silence means no one is transmitting. In AI-assisted development, Radio Silence means shipping AI-generated documentation without reading it. The documentation exists — it looks professional, it uses proper dartdoc syntax, it appears in IDE tooltips. But nobody verified whether it describes the actual code.

**How it happens:**

1. You prompt the AI: "Generate dartdoc comments for this file"
2. The AI produces clean, professional documentation for every class and method
3. You commit the file — the code looks well-documented
4. A new team member reads the docs and learns that `login()` "authenticates against the backend API"
5. The new team member spends two hours looking for the API endpoint that does not exist
6. The documentation was wrong. The method checks hardcoded credentials.

**The damage is subtle.** Wrong documentation is worse than no documentation. No documentation forces developers to read the code. Wrong documentation sends them on a wild goose chase, trusting words that do not match reality.

**Common lies in AI-generated dartdoc:**
- "Fetches data from the server" — when the data comes from a local mock
- "Persists the preference to local storage" — when the value is in-memory only
- "Validates the input against the schema" — when there is no schema
- "Thread-safe implementation" — in a single-threaded Dart isolate

Each of these sounds plausible. Each is wrong. The AI generated them because they match common patterns in its training data, not because they match your code.

**The fix:** Treat AI-generated documentation the same way you treat AI-generated code — as a first draft that requires review. Read every comment. Compare it to the code. Fix anything that describes functionality that does not exist.
