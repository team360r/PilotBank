<QuizQuestion
  question="A currency input validator uses RegExp(r'^\d+\.\d{2}$'). What is wrong with it?"
  options={[
    "It allows negative numbers",
    "It requires exactly two decimal places, rejecting valid whole numbers like '50' and single-decimal amounts like '5.5'",
    "It does not check for maximum amount",
    "The regex syntax is invalid in Dart"
  ]}
  correctIndex={1}
  explanation="The regex ^\d+\.\d{2}$ requires a dot followed by exactly two digits. This means '50' is rejected (no decimal point), '5.5' is rejected (only one decimal digit), and '100.0' is rejected. A better approach is to use double.tryParse() for parsing and separate logic for decimal-place limits, rather than a rigid regex."
/>

<QuizQuestion
  question="What is the 'False Horizon' anti-pattern?"
  options={[
    "Writing tests that always pass regardless of the code",
    "Using AI to generate UI that does not match the design mockups",
    "Trusting code that looks correct and works on the happy path without testing edge cases — the code appears level but is subtly banking",
    "Setting unrealistic deadlines based on AI-estimated development time"
  ]}
  correctIndex={2}
  explanation="A false horizon in aviation is when visual cues mislead the pilot into thinking the aircraft is level when it is banking. In development, it is code that compiles, has sensible variable names, follows the right patterns, and works with valid input — but fails on edge cases. The fix is to trust your instruments (tests and edge-case verification) over your eyes (code reading)."

<QuizQuestion
  question="You type '0' into the transfer amount field and the validation accepts it. What is the most likely cause?"

This is a classic false-horizon bug — the validation function exists and looks correct, but misses a key constraint."
