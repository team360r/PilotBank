## Section 1: Anatomy of a Prompt

This is the single most important skill in AI-assisted development: writing prompts that produce the code you actually want.

Watch what happens with a vague prompt versus a precise one.

### The bad prompt

> "make a banking app model"

What comes back? Something like a 200-line `BankingModel` class with fields for account number, routing number, SWIFT code, balance, overdraft limit, interest rate, joint account holder names, branch address... none of which match your actual data requirements. You spend longer deleting the wrong parts than you would have spent writing it yourself.

### The good prompt

> "Create a Dart data model for a Transaction with: id (String), accountId (String), amount (double), description (String), date (DateTime), type (enum: credit/debit). Use const constructor, named required parameters. No JSON serialisation, no factory constructors."

What comes back? Exactly what you asked for. One class, six fields, correct types, correct constructor style, nothing extra.

The difference is not AI quality — it is prompt quality. A precise prompt constrains the AI's output to what you actually need.

### The five parts of an effective prompt

Every good code-generation prompt includes these elements:

List every field and its type explicitly. "**id (String), accountId (String), amount (double)**" leaves no room for the AI to guess. If you leave types out, the AI will choose them — and it will often choose wrong.

Specify patterns and conventions. "**const constructor, named required parameters**" tells the AI how to build the constructor. Without constraints, you get whatever the AI's training data says is most common — which may not match your project.

## Section 2: Comments as Prompts

Here is the most underused prompting technique in AI-assisted development: writing a comment and letting the AI complete the code below it.

Most developers only interact with their AI tool through the chat panel. But inline completions — triggered by comments — are faster for simple, well-defined code generation tasks.

Try this. Open a new Dart file and type:

```dart
// A User model with id (String), name (String), email (String).
// Use const constructor with named required parameters.
```

Then press Enter and wait. Your AI tool should suggest the complete `User` class below the comment. The comment acts as the prompt, and the AI reads it as context for what should come next.

**Why this works so well:**

- The comment stays in your code as documentation
- The AI has the full file context (imports, other classes) alongside your instruction
- You can iterate — adjust the comment, delete the suggestion, and let the AI try again
- It is faster than switching to a chat panel and copying code back

**When to use comments as prompts:**

- Data models and simple classes
- Individual functions with clear input/output
- Enum definitions
- Configuration objects

**When comments are NOT enough:**

- Multi-file generation (the AI needs to see multiple files)
- Complex logic requiring back-and-forth discussion
- When you need the AI to explain its reasoning

## Section 3: Build the Data Models

Time to put prompting into practice. PilotBank needs four data models and two enums. You will create them in `lib/data/models.dart`.

<DualPath exercise="data-models">
  <Manual>

**By hand (~20 min):**

Create `lib/data/models.dart` with the following:

1. An `AccountType` enum with values: `current`, `savings`, `credit`
2. A `TransactionType` enum with values: `credit`, `debit`
3. A `User` class with fields: `id` (String), `name` (String), `email` (String) — all final, const constructor, named required parameters
4. An `Account` class with fields: `id` (String), `name` (String), `type` (AccountType), `balance` (double) — all final, const constructor, named required parameters
5. A `Transaction` class with fields: `id` (String), `accountId` (String), `amount` (double), `description` (String), `date` (DateTime), `type` (TransactionType) — all final, const constructor, named required parameters
6. An `AppNotification` class with fields: `id` (String), `title` (String), `body` (String), `timestamp` (DateTime), `read` (bool, defaults to false) — all final, const constructor, named required parameters except `read`

That is 4 classes, 2 enums, and careful attention to which parameters are required versus optional with defaults. By hand this takes practice with Dart's constructor syntax.

  </Manual>
  <AIAssisted>

**With your copilot (~5 min):**

Prompt for each model one at a time. Do not ask for all four in a single prompt — that gives the AI too much freedom to make structural decisions.

**Prompt 1 — Enums:**

> "Create two Dart enums for a banking app: AccountType with values current, savings, credit. TransactionType with values credit, debit."

**Prompt 2 — User:**

> "Create a Dart data model class called User with fields: id (String), name (String), email (String). All fields final. Use const constructor with named required parameters. No JSON serialisation, no factory constructors, no methods."

**Prompt 3 — Account:**

> "Create a Dart data model class called Account with fields: id (String), name (String), type (AccountType), balance (double). All fields final. Use const constructor with named required parameters."

**Prompt 4 — Transaction:**

> "Create a Dart data model class called Transaction with fields: id (String), accountId (String), amount (double), description (String), date (DateTime), type (TransactionType). All fields final. Use const constructor with named required parameters."

**Prompt 5 — AppNotification:**

> "Create a Dart data model class called AppNotification with fields: id (String), title (String), body (String), timestamp (DateTime), read (bool, default false). All fields final. Use const constructor with named required parameters except read which has a default value."

Assemble all the outputs into a single `lib/data/models.dart` file. Add a doc comment at the top: `/// PilotBank data models.`

  </AIAssisted>
  <CrossCheck>

Review your `models.dart` file against this checklist:

- Both enums are defined as separate top-level types (not nested inside a class)
- All class fields are marked `final`
- All constructors use `const`
- All constructors use named parameters with `required` (except `read` in `AppNotification`)
- `AppNotification.read` has a default value of `false` using `this.read = false`
- `AccountType` uses `current` (not `checking` — we are using British banking terminology)
- No `toString`, `hashCode`, `operator ==`, `toJson`, `fromJson`, or `copyWith` methods were added
- The file compiles: run `flutter analyze lib/data/models.dart`

**Common AI mistakes on this task:**

Adding `toJson`/`fromJson` even when told not to — AI tools are trained on code that almost always includes serialisation. Using `checking` instead of `current` for the account type — the AI defaults to American banking terms. Making `read` a required parameter instead of giving it a default value. Generating a `Notification` class name instead of `AppNotification` — `Notification` conflicts with Flutter's built-in class.

  </CrossCheck>
</DualPath>

## Section 4: Chat vs Inline — When to Use Each

Now that you have used both the chat panel and inline completions (comments as prompts), here is when to use each.

### Inline completions (ghost text)

Best for:
- Single functions or methods
- Completing code you have already started typing
- Small additions to an existing file
- Data models prompted by comments (as you just did)

The key advantage is speed. You stay in your editor, the suggestion appears as you type, and you accept or dismiss without switching context.

### Chat panel

Best for:
- Multi-file code generation
- When you need an explanation alongside the code
- Complex logic that requires back-and-forth refinement
- When you want to see alternative approaches

The key advantage is depth. You can describe what you want in natural language, ask follow-up questions, and have a conversation about trade-offs.

### The decision rule

**If you can describe what you want in a one-line comment, use inline.** If you need a paragraph, use chat.

For the data models exercise, comments-as-prompts (inline) worked well because each model was self-contained and the spec was short. For the mock data, the chat panel was better because the spec was longer and involved multiple interconnected objects.

## Section 5: The Anti-Pattern — Fly-by-Night

Every chapter names an anti-pattern. Chapter 1's is the **Fly-by-Night**.

A Fly-by-Night prompt is one filed without a plan. It is vague, unscoped, and produces output that requires extensive rework — which defeats the purpose of using AI in the first place.

**Signs you are flying by night:**

- Your prompts do not mention data types
- You do not specify constructor style or patterns
- You accept the first suggestion without re-prompting, even when it is not quite right
- You ask for "a model" instead of specifying exactly which fields it needs
- You let the AI decide your app's data structure instead of designing it yourself
- Your prompts end with a period instead of constraints — "Make a user model." versus "Make a User class with id (String), name (String), email (String), const constructor, named required parameters."

**The fix:** Before you type a prompt, write down what you want on paper (or in your head). The fields. The types. The constraints. The things you do NOT want. Then translate that spec into a prompt. If you cannot describe what you want precisely, you are not ready to prompt — you are still designing.
